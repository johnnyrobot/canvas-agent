# Canvas Agent — Deep Review & Improvement Roadmap

**Date:** 2026-06-12 · **Method:** two multi-agent passes (strategic synthesis + a deeper per-subsystem code review), with all load-bearing external facts independently fact-checked. **Status of code claims:** each is cited to `file:line`; the high-severity items were traced by ≥2 independent reviewers. Treat impact/effort (1–5) as informed estimates to validate.

> **Honesty note.** The first workflow was largely killed by a transient server-side rate limit; its surviving synthesizer produced a strong roadmap by reading the real code. A second, targeted gap-fill (this document's main source) ran 9 systematic subsystem reviews + 3 verified research angles. A few agents (contrast engine, render/audit internals, Canvas-API research, some adoption-stat verifiers) were rate-limited and did **not** run — see §10 for what remains unverified. Some adoption percentages came back "uncertain" in fact-checking and are flagged as indicative, not confirmed.

---

## 1. Verdict

The architecture is sound and the security *foundations* are real (read-only Canvas enforcement is airtight; SQL is uniformly parameterized; the allowlist is a true tokenizer→tree→repair pipeline with layered XSS defense; the gate *is* invoked on every emitted fragment). **But the deep review found a cluster of correctness/safety defects that undercut three core promises — "the gate withholds the badge on real failures," "BUILD/REMEDIATE run the deterministic engine," and "Remediate converts documents" — and these should be fixed before most feature work.** The strategic roadmap below is still right; it's been **reordered** so the integrity fixes come first.

## 3. Verified external landscape (the "why now")

- **ADA Title II (regulatory clock).** DOJ 2024 final rule = **WCAG 2.1 AA**. An **Interim Final Rule published Apr 20, 2026** extended deadlines (comment period through Jun 22, 2026): **Apr 26, 2027** for populations ≥50k (effectively all public higher-ed) / **Apr 26, 2028** for smaller. Scope **explicitly includes Canvas course materials.** DOJ's stated rationale was **overestimated institutional capacity** (not, as an earlier draft claimed, "generative-AI limits"). Targeting WCAG **2.2** AA (as the PRD does) *exceeds* the legal floor — keep it, but the honesty banner should reference 2.1 AA as the standard met-and-exceeded. ([UPCEA](https://upcea.edu/doj-extends-accessibility-deadline-to-april-2027-policy-matters-april-2026/), [Reed Smith](https://www.reedsmith.com/articles/doj-extends-digital-accessibility-compliance-dates-under-title-ii-of-the-ada/))
- **Automation coverage (the honesty number).** axe-core finds **~57% of WCAG issues by volume**; only **~29.5% of WCAG 2.2 criteria** are fully automatable (~60% need manual). ([Deque](https://www.deque.com/automated-accessibility-coverage-report/)) WebAIM Million 2024 (**supported** in our check): **96.4%** of detected errors concentrate in a few categories, and pages **with ARIA average +34.2% errors**.
- **The overlay precedent (the moat).** FTC fined **accessiBe $1M (Jan 2025)** for "automatically comply" claims; 800+ overlay users were still sued; courts reject overlays as compliance proof. The unconditional, never-auto-certify gate is the exact opposite — that's the defensibility. ([Lainey Feingold](https://www.lflegal.com/2025/01/ftc-accessibe-million-dollar-fine/))
- **Competitive shifts (new since the PRD).** **UDOIT (open-source Canvas scanner) was removed from Canvas Sep 12, 2025.** **Instructure shipped its own native Course Accessibility Checker (GA Feb 25, 2026, Feature Option)** — scans Pages+Assignments course-wide for alt-text/headings/tables with guided remediation. So "course-wide HTML scanning" is **partly commoditized by Canvas itself**; differentiation = deterministic gating + honesty artifacts + **documents/PDF/Office** + a **Build (generate)** path + **on-device alternative formats** + on-device privacy. Market: Anthology Ally ~83% share (slow growth); YuJa Panorama ~17% but ~48% CAGR; both gate alternative-format generation behind the **cloud**. ([ListEdTech](https://listedtech.com/blog/digital-accessibility-in-higher-education-two-vendors-to-watch/), [Instructure](https://community.instructure.com/en/discussion/664657/course-accessibility-checker-feature-overview))
- **On-device AI stack.** **Ollama 0.19 (Mar 30, 2026) switched to an MLX engine on Apple Silicon** (~2× decode on M-series; unbenchmarked here). Structured JSON-schema outputs are supported. **`sqlite-vec`** gives native in-SQLite vector search with graceful degradation + **hybrid FTS+semantic** — the natural on-device path for SB8. Apple's Foundation Models framework (macOS 26, `@Generable`) is an alternative but Swift-only. ([Ollama](https://ollama.com/blog), [sqlite-vec](https://github.com/asg017/sqlite-vec))

## 6. Research-driven design principles (verified-evidence-backed)

1. **Per-issue DISPOSITION taxonomy** (replaces the flat pass/fail): `deterministic-auto-fixed` (lang, page title, list/heading nesting, scope on *simple* tables, accepted contrast suggestion) · `draft-needs-approval` (alt text, link/button name, captions/transcripts) · `human-only-notVerifiable` (reading order, complex/merged tables, color-as-information, focus order, reflow). Encodes the machine-fixable/human-judgment line directly into the gate + badge logic. (`gate.ts`)
2. **NO-ARIA-SYNTHESIS guardrail (new anti-goal).** The repair engine must prefer native HTML semantics and **never** emit `role=`/`aria-*` to "fix" content; the auditor should down-rank orphaned/contradictory ARIA. (WebAIM 2024 **supported**: ARIA pages average +34.2% errors.)
3. **Derive, don't invent, accessible names.** Empty-link (44.6%) / empty-button (28.2%) are *name* problems — fix deterministically from existing visible text/title/adjacent label; if none unambiguous, route to `draft-needs-approval`. Zero hallucination.
4. **Decouple the motivational score from the conformance gate.** A per-page "Automated Checks" score that can legitimately reach green/100 (the best-evidenced faculty behavior lever, per the Ally studies — *percentages uncertain in our check*) must be **architecturally separate** from the never-auto-certify panel, or the honesty banner kills the dopamine loop. An unconfirmed AI draft holds the page in "needs your review," not green.
5. **Role-targeted human-review handoff.** Let faculty bundle/assign the human-only items (alt text, captions, color-meaning, complex tables, focus order) to an instructional designer / disability-services role (responsibility is shared, not solely faculty).
6. **JIT coaching.** Wire GUIDANCE as issue-triggered, deep-linked "why this matters + how to fix in Canvas" from each finding — not a separate mode users must remember.
7. **ASR caption gate.** An auto-generated transcript/caption can **never** clear a media caption blocker (real ASR WER 8–20% even at "99%" claims); show the WER gap and ship a lightweight in-app caption editor.
8. **Complex-table classifier.** Auto-apply `scope=` only on simple-uniform tables; merged/multi-level/offset → `notVerifiable` id/headers item, never auto-`scope`. Same classifier governs Build templates.
9. **ThemeResolver as a confirm-required suggester.** Compute the nearest WCAG-AA color (minimal perceptual delta) and *suggest* it; never silently rewrite a brand color. Add a distinct WCAG 1.4.1 "use of color" check (which contrast math cannot detect).
10. **Progressive first-scan.** Surface deterministic wins + a provisional score immediately; run the heavier Playwright+axe render in the background (time is the #2 faculty barrier).

## 7. Anti-goals / honest limits

Never present a clean automated scan as legal compliance (mandatory, non-removable coverage banner + badge-withholding). No "fix everything" button. Keep alt-text quality, captions, color-as-information, complex tables, focus order, reflow as human-reviewed. **Never synthesize ARIA.** An issue id disappearing ≠ fixed (could be hidden/occluded/false-negative). Resist Canvas write-back (abandons the read-only thesis) and cloud (on-device is a privacy guarantee).

## 9a. Auditor internals — reviewed 2026-06-13

The deterministic auditor (the §10 "review it next" item) has now been code-reviewed. Verdict: **contrast math is correct; the axe ruleset had a real coverage gap (now fixed); the render/extraction layer has fail-open approximations the coverage banner must disclose.**

- 🚨→✅ **WCAG coverage gap (FIXED).** `AXE_TAGS` was `[wcag2a, wcag2aa, wcag21aa]` — it omitted `wcag21a` (all WCAG 2.1 Level A rules) and every `wcag22*` tag, so despite the "WCAG 2.2 AA" claim the scan never ran 2.1 Level A or 2.2 rules (e.g. 2.5.8 target-size, tagged `wcag22aa`). Now `[wcag2a, wcag2aa, wcag21a, wcag21aa, wcag22aa]` (axe-core 4.12 has no `wcag22a`). `best-practice`/AAA stay excluded (correct — not AA failures). Locked by `playwright-runner.test.ts`. `engine/render/playwright-runner.ts:27`
- ✅ **Contrast math is correct.** WCAG sRGB linearization (0.03928 / 2.4γ), luminance coefficients (0.2126/0.7152/0.0722), `(L+0.05)/(darker+0.05)`, 148 named colors, hex 3/4/6/8, rgb/rgba comma+slash; `transparent` throws fail-safe. `engine/contrast.ts`
- ⚠️→✅ **Alpha compositing (FIXED).** The merged contrast-adjudication feature composites translucent layers (`parseColorAlpha` + `compositeLayers`) instead of scoring at full opacity. Locked by the opt-in browser integration ("text over a translucent overlay is composited and flagged"). `engine/render/run-contrast.ts`, `contrast.ts`
- ⚠️→✅ **Gradient / background-image detection (FIXED).** The browser-side extractor now classifies each text run's background as `layers` / `gradient` / `image` / `unresolvable` and samples image pixels behind the text; gradients/images are adjudicated or routed to review rather than assumed white. Locked by integration ("text over a low-contrast gradient … flagged", "background image yields a contrast warning"). `engine/render/playwright-runner.ts` (`EXTRACT_TEXT_RUNS`), `sample.ts`, `png.ts`
- ⚠️→✅ **Shared Canvas shell (FIXED).** Audit and preview/export now consume ONE pure source-of-truth module (`engine/render/canvas-shell.ts`); the audit renders the full Canvas CSS (links `#0374b5`, tables, buttons, blockquote) the user sees, so "what you see matches what was audited" now holds. `engine/render/playwright-runner.ts` + `app/renderer/preview.ts` → `canvas-shell.ts`. Locked by `canvas-shell.test.ts` (byte-parity).
- ⚠️ **One Chromium launch per fragment** (`run()` launches+closes a browser each call) — fine per-page, but reuse a browser/context across a course-wide scan (SB1). (Open.)
